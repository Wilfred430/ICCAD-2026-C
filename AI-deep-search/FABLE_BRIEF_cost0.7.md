## 1. 問題定義

- **輸入**：一個晶片 case，含 21～120 個矩形模組（block）。每個 block 有：
  - 目標面積 `area`（soft block 可自由選長寬比，只要面積誤差 ≤ 1%）。
  - 或是 **fixed-shape**（長寬鎖死）/ **preplaced**（連位置都鎖死）——這兩類是硬約束。
  - 網表連線：block↔block (b2b)、pin↔block (p2b)，帶權重。
  - 外部固定腳位 terminal 的座標。
- **輸出**：每個 block 的 `(x, y, w, h)`，要求：不重疊、不超出、滿足所有硬約束。
- **目標**：最小化 Cost（見 §2）。核心是「面積小 + 線長短（HPWL）+ 不違反約束 + 跑得快」。

**軟約束（violation 會扣分但不致命）**：
- **Grouping**：同群組的 block 必須連成一塊（每個成員至少跟一個同群夥伴貼邊）。
- **MIB**：同 MIB 群組的 block 必須共享同一組 `(w,h)`。
- **Boundary**：指定要貼某條邊界（L/R/T/B）或角落的 block 要貼到位。

## 2. 評分公式 + 「0.7」的精確數學拆解（本文件最重要的一節）

Cost 公式（已對原始碼逐字驗證，常數不可改）：

```
Cost = Q · P · R           （若不可行 feasible=false，直接 Cost = 10）

Q = 1 + 0.5 · (HPWL_gap + Area_gap)      ← 品質項
P = exp(2 · V_rel)                        ← 軟約束懲罰項
R = max(0.7, RT^0.3)                      ← runtime 項
```

其中：
- `HPWL_gap = (你的HPWL − baseline HPWL) / baseline HPWL`，**有號**。負值 = 比 baseline 好。
- `Area_gap = (你的bbox面積 − baseline面積) / baseline面積`，**有號**。
- `V_rel = (V_group + V_mib + V_boundary) / N_soft ∈ [0, 1]`。所以 `P ∈ [1, e² ≈ 7.39]`。
- **`RT = 你的runtime / 該 test case 所有參賽隊伍 runtime 的中位數`**（spec 原文
  footnote：「computed independently for each test design, using that
  individual test case's median runtime as the sole reference point」）。
  這是**跨隊伍、逐 case 各自獨立計算的中位數**，事前無法得知、也無法在本地
  精準瞄準——是個隨所有參賽者表現變動的移動標靶。跑得越快 `RT` 越小、`R`
  越低，下限硬卡在 0.7（最多省 30%）；跑得慢則 `R` 無上限暴漲。實務上唯一
  能做的是「盡量壓低每個 case 的絕對耗時」，沒有能精準命中的數字目標。
- `baseline` = **資料集本身自帶的 ground-truth（近似最優）解**，不是某個
  標準 SA baseline 跑出來的參考解（已用官方 `iccad2026_evaluate.py::
  _extract_baseline()` 原始碼 + 驗證集 metrics 交叉核對確認）。

### 「Cost = 0.7」到底要求什麼？（把三項拆開來看）

因為 `R` 的最小值是 0.7、`P` 的最小值是 1，所以對任意品質項 Q，Cost 的下限是 `Q · 1 · 0.7 = 0.7Q`。

要讓 `Cost ≤ 0.7`，必須同時滿足**三個獨立條件**：

| 條件 | 數學要求 | 白話 | 難度 |
|---|---|---|---|
| **① 零軟約束違規** | `V_rel = 0` → `P = 1` | grouping / MIB / boundary **一個都不能違反** | 極高（見下） |
| **② 極快** | `RT ≤ 0.7^(1/0.3) ≈ 0.305` → `R = 0.7` | runtime ≤ **該 case 所有隊伍 runtime 中位數的 ~30%**（移動標靶，見上） | 中 |
| **③ 品質追平 baseline** | `hpwl_gap ≤ 0` 且 `area_gap ≤ 0` → `Q = 1` | 線長與面積都不輸 baseline（贏了也沒獎勵，見下） | 中 |

> **⚠️ 2026-07-08 重要訂正（gap 有 clamp）**：官方
> `iccad2026_evaluate.py::compute_cost` 第 322 行是
> `quality = 1 + 0.5·(max(0, hpwl_gap) + max(0, area_gap))`——每個 gap 都被
> `max(0,·)` **夾住**。所以 **Q 一律 ≥1，贏過 baseline 完全不加分**（gap 為負
> 也被夾成 0）。這代表 0.7 是**真正的地板**（`1×1×0.7`），不可能靠品質壓破。
> 此前本文件誤寫「品質小贏可壓破 0.7 到 ~0.6」是錯的，已刪除。

> **這個拆解顛覆直覺**：壓到 0.7 的關鍵**不是**把線長/面積打得比 baseline 好很多（打得再好也只夾到 Q=1），而是：
> - **①「完美 feasibility（連軟約束都零違規）」是真正的瓶頸。** 量化一下：假設某大 case 有 `N_soft = 40` 個軟約束槽位，只要**違反一個**，`V_rel = 1/40`，`P = exp(2/40) = exp(0.05) ≈ 1.051`，Cost 立刻從 0.7 變 0.735，直接出局。所以是**零容忍**。
> - **② runtime 要壓到跟自己競爭的隊伍們的中位數的 1/3**——事前不知道那個中位數是多少，唯一能做的是盡量壓低絕對耗時；這仍然強烈偏好「攤提式 ML 推論 / 解析式擺放」而非「每個 case 從頭跑昂貴搜尋」，因為那類方法的絕對耗時本來就低，不管中位數落在哪都有安全邊際。
> - **③ 品質只要「追平」baseline 即可**（`gap ≤ 0` → Q 夾到 1）。因為有 clamp，贏過 baseline 一點好處都沒有，所以品質的最佳投報是「剛好追平就收手」，多餘力氣全投①②。

**所以真正的戰場是：如何在「不輸 baseline 品質 + 30% 的時間預算」內，做到『任何 n、任何約束組合都零軟違規』。** 這是我要 Fable 突破的核心。

### e^(n/12) 總分加權（決定資源分配）

總分是各 case 以 `e^(n/12)` 加權相加（n = block 數，21～120；已對照 spec 原文
Total Score 公式確認，**不是** `e^n`——本地 `iccad2026_evaluate.py` 的
`compute_total_score()` 用純 `e^n`，跟 spec 不符，那支腳本有 bug，spec 才是
權威）。一個 n=120 的 case 權重是 n=21 的 `e^(99/12) = e^8.25 ≈ 3820` 倍。
**大 case 仍決定性地重要，但 n≈60–90 的中段 case 也有實質分數貢獻，不是只有
n≈90–120 存在。** 任何方案在大 n 上不能垮，但也不該完全放棄中段 case。

## 3. 現狀：試過什麼、分數多少

我有三條並存路線：

- **A（主力）B*-tree + Fast-SA**（C++、多執行緒多 seed）：離散拓樸搜尋。成熟穩定，但見 §5 的搜尋空間牆——大 n 的品質受限。
- **B（ML warm-start）Graph-Transformer 回歸座標**：直接回歸 `(cx,cy,w,h)`。**已診斷失敗**：多峰解導致 mode collapse（兩個合法解的平均 = 不合法解，MSE 訓練會去取平均）。
- **C（電靜力法 / ePlace 風格）**：純 Python，把 block 當帶電粒子做連續梯度下降 + legalize + repair。**目前綜合最佳**，100% feasible，但單 case Cost 平均仍明顯 > 1。

**近期新做的（生成式拓樸模型）**：訓練一個 autoregressive Transformer**逐步生成 B*-tree**（用 pointer network 選「下一個 block 接在哪」），cross-entropy 訓練繞開 mode collapse。用主辦方 1M 訓練集裡的近似最優解 `tree_sol` 當示範。
- 現況：parent-pointer 準確率 87%，採樣拓樸 100% feasible（無重疊），**但單 case Cost 仍 ≈ 5.35**。
- 為什麼還這麼高：(a) soft block 長寬目前用佔位正方形，沒優化長寬比；(b) 幾何精修的四道 repair pass 還沒接上；(c) 只有監督式預訓練，**還沒做「用真實 Cost 當獎勵」的強化微調**。

## 4. 硬約束與陷阱（違反 = Cost 10，一翻兩瞪眼）

- **HPWL 是「中心點對中心點」的加權曼哈頓距離**：`cx_i = x_i + w_i/2`。不是 bbox 半周長。
- **fixed-shape / preplaced 是硬約束**：任何偏差 = Cost 10。它們住在 packer（不可變長寬、錨定座標），不進軟約束項。
- **soft block 面積容忍度 1%**：`|w·h − area| / area ≤ 0.01`。放寬 = 本地過、官方掛。
- **boundary code 是 bitmask**：1=左, 2=右, 4=上, 8=下，角落用加總（如 5=左上）。
- **不可逆向工程主辦方的資料生成器**（規格明令，會取消資格）。

## 5. 搜尋空間的數學牆（為什麼純搜尋在大 n 贏不了）

- n=120 的 B*-tree 拓樸組合數 ≈ `10^250`。
- SA 在時限內能評估的解 ≈ `10^6`。
- 搜到的比例 ≈ `10^-244`——等於在太平洋裡憑運氣撈一個特定水分子。
- **結論**：大 n 靠純搜尋（不管 SA 調多快）在數學上不可能穩定逼近最優。必須靠「從 1M 資料學到的先驗」把有意義的搜尋範圍壓縮好幾個數量級。這也呼應 §2 條件②：與其每個 case 花大量時間搜尋，不如攤提式 ML 推論一次到位。
